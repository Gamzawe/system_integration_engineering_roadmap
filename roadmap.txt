# 24-Month .NET Backend and Systems Integration Roadmap

**Weekly load:** 10 hours
**Target outcome:** become a **strong junior-to-mid .NET backend / systems integration engineer**, with realistic potential for **mid-level roles** if your portfolio and real work experience are strong.

## Your starting advantage

This roadmap is adjusted for your profile:

*   **Completed frontend roadmap** — you already know Git, IDE tools, HTTP, REST, JSON, debugging, and JavaScript/TypeScript fundamentals.
*   **Active biometric SDK integration work** — you already write production integration code, handle device communication, and understand real-world SDK challenges.
*   **Months 1 and 5 are compressed** to skip what you already know and reinvest that time into protocol diversity, data mapping, and observability — skills that separate integration engineers from generic backend developers.
*   **Start light job applications by Month 16**, not Month 22. Your real-world experience makes you employable earlier than a pure self-taught candidate.

## Execution rules

Use this roadmap as a **structured guide**, not a prison.

That means:

*   aim to finish the **week outcome**
*   do not panic if one "day" takes two days
*   debugging is part of learning, not delay
*   always favor **building and documenting** over endless watching
*   **document your real biometric work patterns alongside this roadmap** — real-world integration experience is your biggest competitive advantage

## Weekly study pattern

Use this every week:

*   **4 hours learning**
*   **4 hours building**
*   **2 hours review, refactor, notes, README, and Git**

## Monthly success rule

Every month should produce:

*   **1 working deliverable**
*   **1 cleaned GitHub push**

# Phase 1: Core Programming and Developer Foundations

## Month 1: C# Crash Course and First Project (Accelerated)

You already know Git, an IDE, and basic programming from your frontend work. This month focuses on **C# language specifics** and getting productive fast.

### Week 1: C# Syntax Sprint

**Day 1 (Setup):** Install Visual Studio Community or VS Code, install the .NET SDK, verify with `dotnet --version`, create a GitHub repo with `.gitignore` for .NET.

**Day 2 (C# Types):** Run `dotnet new console`. Learn `string`, `int`, `bool`, `double`, `DateTime`, `int.Parse()`, and `int.TryParse()`. Note differences from JavaScript/TypeScript.

**Day 3 (Control Flow):** `if/else`, `switch`, `while`, `for` — same concepts as JS but learn C# syntax and pattern matching.

**Day 4 (Exceptions):** Learn `try/catch/finally`. Note: C# exceptions are stricter than JS. Practice `InvalidOperationException`, `ArgumentException`, `FormatException`.

**Day 5 (Mini Project):** Build a kiosk-style console app with menu, input validation, and exception handling.

### Week 2: Methods, Debugging, and C# Specifics

**Day 1 (Methods):** Create methods with parameters, return values, `out`, and `ref`. Learn method overloading (not available in JS).

**Day 2 (Value vs Reference):** Learn the difference between value types (`int`, `struct`) and reference types (`class`, `string`). This is critical in C# and different from JS.

**Day 3 (Nullability):** Learn nullable types (`int?`), null-conditional (`?.`), null-coalescing (`??`), and nullable reference type annotations. C# takes nulls more seriously than JS.

**Day 4 (Debugging):** You know browser DevTools. Now learn Visual Studio breakpoints, step over, step into, watch variables, and the Immediate Window.

**Day 5 (Project Commit):** Refactor the kiosk app into clean methods, add a proper README, and push.

### Week 3: Enums, Structs, and Type Safety

**Day 1 (Enums):** Create enums for device status, scan result types. Learn `[Flags]` enums.

**Day 2 (Structs):** Learn when to use `struct` vs `class`. Create a `ScanCoordinate` or `DeviceAddress` struct.

**Day 3 (Generics Intro):** Learn `List<T>`, `Dictionary<TKey, TValue>`, and why C# generics are stricter than TS generics.

**Day 4 (String Handling):** String interpolation, `StringBuilder`, `string.Format()`, `Span<char>` basics.

**Day 5 (Mini Exercise):** Extend the kiosk app with enums for menu options and scan results.

### Week 4: Functional Patterns and Month Consolidation

**Day 1 (Delegates and Lambdas):** Learn `Action`, `Func`, and lambda syntax. Compare to JS arrow functions.

**Day 2 (Events):** Learn the event pattern in C#. Create a `ScanCompleted` event on the kiosk.

**Day 3 (Tuples and Records):** Learn tuple returns and `record` types for immutable data.

**Day 4 (Pattern Matching):** Learn `is`, `switch` expressions, and property patterns.

**Day 5 (Month Review):** Push a polished kiosk app demonstrating C#-specific features. Write a README section comparing C# patterns to their JS/TS equivalents.

### Month 1 Output

* 1 working console app using C#-specific features (enums, events, records, nullability)
* 1 GitHub repository with README
* clear understanding of how C# differs from JavaScript/TypeScript

### Week 47: Offline Thinking, Local Persistence, and Windows Services Deployment

**Day 1 (Problem):** Study what happens if the API is unavailable.

**Day 2 (Fallback Storage):** Save failed submissions locally.

**Day 3 (Recovery Logic):** Add resend logic.

**Day 4 (Windows Services Deployment):** Integration engineers in practice deploy software as Windows Services running on-site kiosk machines, not in Docker on day one. Learn the full deployment cycle for a .NET Worker Service as a Windows Service: use `dotnet publish -c Release --self-contained true -r win-x64 -p:PublishSingleFile=true` to produce a single executable. Install it with `sc.exe create MyService binPath= "C:\path\to\service.exe"`. Start with `sc.exe start MyService`, stop with `sc.exe stop MyService`. Configure automatic restart on failure using `sc failure MyService reset= 0 actions= restart/5000`. Understand how to update a running service safely: stop → replace binary → start. Add `UseWindowsService()` to `Program.cs` so the same codebase runs as both a console app in dev and a Windows Service in production. This is how your real biometric agents likely run today — formalizing this knowledge matters.

**Day 5 (Review):** Document offline-first behavior and add a Windows Service deployment checklist to your README.

### Week 48: Data Mapping, Format Diversity, and Year 1 Consolidation

**Day 1 (Data Mapping Theory):** Learn why integration systems must map between vendor-specific and canonical data models. Most of your integration work will involve transforming data from one format to another.

**Day 2 (Canonical Models):** Design canonical models for your system (e.g., `CanonicalScanResult`, `CanonicalDeviceStatus`) that are vendor-agnostic. Map vendor-specific DTOs to these using manual mapping (prefer explicit mapping over AutoMapper for complex integration logic — it is easier to debug and maintain).

**Day 3 (Non-JSON Formats):** Integration systems rarely deal with JSON alone. Parse XML using `XDocument` and `XmlSerializer`. Read and write CSV files with `CsvHelper`. Handle fixed-width text formats with `string.Substring()` or `Span<char>`. Many legacy devices, ERP systems, and enterprise partners send XML or CSV, not JSON.

**Day 4 (Architecture Review):** Draw system components. Document integration layer, logging, worker, persistence, data mapping, and format adapters. Remove hacks and duplication.

## Docker CLI Primer (Before Phase 5)

Month 13 uses Docker to run RabbitMQ. Before starting Month 13, spend a few hours on these Docker basics so you are not learning Docker and messaging simultaneously.

**Concepts:** Image = blueprint. Container = running instance. `docker run` creates and starts a container. `docker ps` lists running containers. `docker logs <name>` reads container output. `docker stop <name>` stops gracefully. `docker rm <name>` removes stopped containers.

**Commands to practice:**

*   `docker run -d --name rabbit -p 5672:5672 -p 15672:15672 rabbitmq:3-management` — run RabbitMQ with management UI
*   `docker ps` — confirm it is running
*   `docker logs rabbit` — read startup logs
*   `docker stop rabbit && docker rm rabbit` — clean up
*   `docker run -d -e ACCEPT_EULA=Y -e SA_PASSWORD=YourPass -p 1433:1433 mcr.microsoft.com/mssql/server:2022-latest` — run SQL Server in a container

**Volume basics:** `docker run -v C:/data:/var/opt/mssql/data ...` persists data across container restarts. Without a volume, all container data is lost on `docker rm`.

**Why this matters now:** For the next 3 months you will use Docker to run RabbitMQ and SQL Server locally without installing them. In Month 15 you will write your own Dockerfiles. For now, just consume existing images.
